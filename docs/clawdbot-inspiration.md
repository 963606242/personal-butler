# Clawdbot 借鉴与结合：个人管家能学到什么

Clawdbot 是近期流行的「把事做完的 AI 助手」：本地部署、持久记忆、多消息平台接入（WhatsApp / Telegram / Discord 等）、技能即 Markdown、主动推送晨报/提醒。本文梳理其核心设计，并对照本产品（个人管家）给出可借鉴、可结合的方向。

---

## 一、Clawdbot 在做什么（简要）

| 维度 | Clawdbot | 个人管家现状 |
|------|----------|--------------|
| **定位** | 把事做完的 AI 助手（执行 > 聊天） | 本地优先的生活管理 + 内嵌 AI 建议 |
| **记忆** | 本地 Markdown（SOUL.md、USER.md、memory/） | 本地 SQLite，有用户/习惯/日程/倒数等 |
| **入口** | WhatsApp / Telegram / Discord / Slack 等 | 桌面 Electron 应用 + **本地 API（api-bridge）** |
| **技能** | Markdown 文件描述工具与流程 | `docs/api-bridge.md` 等文档描述接口 |
| **主动** | Cron 晨报、Webhook 提醒 | 仪表盘今日建议、早报/晚报、倒数日提醒 |
| **数据** | 本地工作区，可 Git 版本控制 | 本地 DB，可选 API 供外部调用 |

共同点：**本地优先、数据在己、可被 AI/脚本调用**。差异：Clawdbot 强在「消息入口 + 执行型智能体」，我们强在「结构化生活数据 + 现成 UI」。

---

## 二、可借鉴的六点

### 1. 「把事做完」而不只是「给建议」

- **Clawdbot**：用户说「加一条明天 10 点开会」，助手直接写日历；说「今天跑步打卡」，助手记一笔。
- **可借鉴**：在现有「获取今日建议」之外，增加**一键执行**：  
  - 从建议文案中解析出「创建日程」「习惯打卡」等意图，经用户确认后调用现有 API（或本应用内接口）直接写入日程/打卡。  
- **结合点**：AI 助手页或仪表盘「今日建议」卡片上增加「执行」按钮，调用 `api-bridge` 或内部服务完成创建日程/打卡，而不是只展示文案。

### 2. 持久记忆 / 用户画像供 AI 使用

- **Clawdbot**：USER.md、SOUL.md、memory/ 让 AI 记住偏好与上下文。
- **可借鉴**：把「当前用户资料、习惯名称、近期日程、倒数日」等做成**结构化上下文**，在调用大模型前注入（或通过 api-bridge 的「摘要」接口暴露）。  
- **结合点**：已有「个人资料摘要」类接口时，在 AI 建议、今日计划等接口里默认带上这些摘要，让模型输出更贴合「这个人」的习惯与安排。

### 3. 技能 = 文档，工具 = API

- **Clawdbot**：技能即 Markdown，描述怎么用工具；代理按文档调工具。
- **可借鉴**：把 `api-bridge` 的接口说明视为「技能文档」，并**结构化**：  
  - 在 `docs/api-bridge.md` 或同类文档中写清：在什么场景下调用哪个接口、入参示例、典型用法。  
  - 若有外部智能体（如 Clawdbot、自建 Bot）要联动，只需让它读这份文档 + 调用本地 API即可「学会」用个人管家。  
- **结合点**：保持并强化「API 说明」即「技能说明」的写法，必要时增加「场景示例」小节（如：晨报场景下先 GET 日程再 GET 习惯打卡）。

### 4. 主动推送：定时任务 + 摘要

- **Clawdbot**：Cron 触发晨报（日历 + 邮件 + 新闻），Webhook 触发事件提醒。
- **可借鉴**：  
  - **定时**：在应用内或通过外部 cron 定时调用本地 API，获取「今日日程 + 习惯概览 + 天气/新闻摘要」，生成一条晨报/晚报文案，再由其他渠道（如 Telegram Bot）发送；或个人管家在桌面端做「每日定时通知」。  
  - **事件**：重要倒数日、习惯连续 N 天等，除应用内提醒外，可暴露 webhook 或轮询接口，供外部系统发推送。  
- **结合点**：api-bridge 已有日程、习惯、倒数等读接口；可增加一个「今日摘要」或「晨报数据」聚合接口，便于外部做定时晨报；桌面端可增加「每日提醒时间」设置 + 本地通知。

### 5. 多渠道入口：消息即入口

- **Clawdbot**：用户通过 WhatsApp / Telegram 等发一句话，助手执行并回复。
- **可借鉴**：个人管家不一定要自己做 IM，但可以**当好「数据后端」**：  
  - 外部 Bot（如 Telegram Bot、Clawdbot 自建技能）通过 api-bridge 读写日程、习惯、倒数，实现「在聊天里加日程 / 打卡」；  
  - 在文档中说明「如何用 Clawdbot / 脚本调用本应用 API」，形成「消息端 + 个人管家数据」的协作。  
- **结合点**：在 `api-bridge.md` 或新文档中增加「与 Clawdbot / 其他 Bot 联动」示例：如何配置 Clawdbot 技能调用本机 3847 端口、示例 prompt（如「帮我往个人管家加一条日程」）。

### 6. 本地优先 + 透明可控

- **Clawdbot**：数据在本地、技能可编辑、记忆可 Git。
- **可借鉴**：  
  - 继续强调「数据不出本机」、API 仅 localhost；  
  - 若有「管家记忆」或偏好（如常用习惯名、默认提醒时间），可考虑以可读格式（如 JSON/Markdown）导出或存放，方便用户查看和备份。  
- **结合点**：设置页或帮助中明确写出「数据存储位置」「API 仅本机」；未来若做「记忆/偏好」文件，可提供导出或路径说明。

## 三、可落地的结合优先级（建议）

| 优先级 | 方向 | 动作示例 |
|--------|------|----------|
| **P0** | 文档与生态 | 在 `api-bridge.md` 增加「与 Clawdbot / 脚本联动」示例；补充「晨报/摘要」聚合接口说明或实现。 |
| **P1** | 一键执行 | AI 建议/今日计划中解析「创建日程」「习惯打卡」，经确认后调用内部或 api-bridge 写入。 |
| **P2** | 主动推送 | 提供「今日摘要」API；或应用内「每日提醒时间」+ 本地通知，内容来自日程/习惯/倒数。 |
| **P3** | 记忆/画像 | 在 AI 调用链中注入用户资料与近期习惯/日程摘要，使建议更个性化。 |

---

## 四、小结

- **学到什么**：Clawdbot 的成功点在于「执行型 AI + 本地记忆 + 消息入口 + 主动推送 + 技能即文档」，和我们的「本地数据 + 本地 API」天然契合。  
- **借鉴什么**：强化「把事做完」（执行写入）、「技能即 API 文档」、主动推送与摘要、以及「消息端 + 个人管家」的联动方式。  
- **结合什么**：在不大改架构的前提下，优先做：**api-bridge 文档与示例增强**、**AI 建议一键执行**、**今日摘要/晨报数据接口或应用内定时提醒**，再视需要做「管家记忆」与更多 Bot 联动示例。

以上内容可作为产品迭代与技术排期的参考，不必一次全做，按优先级逐步结合即可。
